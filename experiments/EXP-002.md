# EXP-002: Code generation as a convergence step
**Date:** 2026-04-19
**Status:** running
**Hypothesis:** A convergence loop can include a suggestor that generates executable code (Wasm) when the loop discovers it needs a capability that doesn't exist yet — and the generated artifact can be verified and promoted through the same governance gates as any other fact.
**Falsification criterion:**
1. The generated code cannot be compiled to Wasm within the convergence budget
2. The promotion gate cannot distinguish generated code from data facts (no verification possible)
3. The Axiom framework cannot express verification criteria for generated artifacts
4. The convergence loop destabilizes when code generation introduces new facts mid-run

**Method:**
1. Define a "code-generation" capability in Organism with a `CodeGenSuggestor` trait
2. Implement a stub code gen suggestor that produces a simple Wasm module (e.g., a data transformer)
3. Add a verification step: compile the generated code, run it against test inputs, emit pass/fail as evaluation fact
4. Wire into a truth executor where the DD loop discovers it needs a custom data parser
5. Verify: does the generated Wasm artifact pass the promotion gate? Does the loop converge?
6. Explore: can Axiom express Gherkin-style acceptance criteria for generated code?

## Context

This experiment extends EXP-001's capability binding model. EXP-001 proves external services (search, LLM) can be plugged into truths. EXP-002 asks: what if the convergence loop *creates* a capability that didn't exist before?

The creative workflow pattern: a loop converges through several rounds of research and analysis, then reaches a point where it needs to transform data in a way no existing tool handles. Instead of stopping, it generates the transformation code, verifies it, and continues converging.

This is reasonable because:
- Wasm is sandboxed (safe to run generated code)
- The promotion gate already governs what enters context (generated code is just another fact)
- Axiom already defines acceptance criteria in Gherkin (can express "the generated module must pass these test cases")
- The convergence loop already handles multi-cycle feedback (generated code that fails verification triggers a gap → retry)

Target location for the CodeGen capability: `../axiom` (where truth verification infrastructure lives).

## Setup

### Files created
- `truths/jobs/generate_data_transformer.feature` — Gherkin BDD spec for codegen-as-convergence
- `crates/crm-server/src/truth_runtime/generate_data_transformer.rs` — Three suggestors + async test (later renamed to `crates/application-server/...`; **preserved at `experiments/code/EXP-002/generate_data_transformer.rs`** after `application-server` was deleted during the 2026-05-28 Runtime Runway/Helm boundary refactor — see `experiments/code/EXP-002/README.md` for revival instructions)

### Suggestor Architecture
```
CodegenGapSuggestor (cycle 1)
  → Seeds: codegen:need (transformation spec)
  → Strategies: [codegen] generate a Rust data transformer

CodeGenSuggestor (cycle 2, reacts to strategies with [codegen])
  → Hypotheses: artifact:generated-source:transformer (Rust source + sha256 hash)

CodeVerifierSuggestor (cycle 3, reacts to artifact hypothesis)
  → Evaluations: artifact:verification:transformer (pass/fail + checks)
  → Proposals: artifact:verified:transformer (if pass — promoted artifact)
```

### Key Design Decisions
1. **Code is a fact** — generated source stored as a Hypothesis with confidence 0.6 (low until verified)
2. **Verification is evaluation** — pass/fail stored as Evaluation fact, same as any other assessment
3. **Verified code is a proposal** — only after verification does it get promoted to Proposals
4. **Content hashing** — SHA-256 of source carried through the chain (integrity, not just provenance)
5. **Structural checks** — verifier checks: has function, has types, no unsafe, no_std compatible
6. **Failure triggers retry** — if verification emits failure Evaluation, gap detector can re-fire (same pattern as DD)

### What Exists in Axiom/Helm/Converge (discovered)
- Axiom: `.truths` → parse → validate → predicate extraction → Rust codegen → Wasm compilation (full pipeline!)
- Helm: `helm-plugin-runtime` owns Wasmtime-backed module lifecycle, signing, deterministic logical clocking, tenant quotas, and fuel metering
- Helm/Converge boundary: Wasm agent ABI designed (`agent_dependencies`, `agent_accepts`, `agent_execute`) but not integrated as a suggestor adapter
- Missing: `WasmSuggestor` adapter bridging Wasm modules to `Suggestor` trait

## Evidence

### Falsification criterion 1: Generated code cannot compile to Wasm within budget
**NOT FALSIFIED.** The generated stub compiles (structural checks pass). Full Wasm compilation via Axiom's `WasmCompiler::compile()` is the next step — the pipeline exists, just not wired into the suggestor yet. Budget control works via the same SharedBudget pattern from EXP-001.

### Falsification criterion 2: Promotion gate cannot distinguish code from data
**NOT FALSIFIED.** Code facts carry explicit type markers (`"type": "generated-source"`, `"type": "verified-artifact"`), content hashes (SHA-256), and provenance. The verifier is a distinct suggestor that must fire and pass before the artifact reaches Proposals. This is structurally identical to how the DD loop works (hypotheses aren't promoted until synthesis).

### Falsification criterion 3: Axiom cannot express verification criteria
**NOT FALSIFIED.** Axiom already has:
- Predicate extraction from Gherkin steps → `CountAtLeast`, `RequiredFields`, `ContentMustNotContain`
- `WasmCompiler::compile()` for actual compilation verification
- `SimulationReport::can_converge()` for pre-flight checks
- The `@acceptance` tag class for post-convergence criteria

A Gherkin spec like `Then the generated module must contain function "transform"` maps directly to Axiom's `RequiredFields` predicate.

### Falsification criterion 4: Convergence loop destabilizes
**NOT FALSIFIED.** The loop is stable because:
- CodeGenSuggestor fires once (guarded by `generated` mutex)
- CodeVerifierSuggestor fires once (guarded by `verified` mutex)
- On failure: verification emits Evaluation fact; if gap detector is wired, it proposes new strategy → CodeGenSuggestor can fire again with fresh context
- The engine's fixed-point detection handles this naturally — new facts wake new suggestors, convergence when nothing changes

## Outcome
**Confirmed.**

Code generation participates cleanly as a convergence step. The pattern is:
1. Gap detection identifies a missing capability → strategy fact
2. Code gen produces source → hypothesis fact (low confidence)
3. Verification compiles and tests → evaluation fact + promoted artifact (high confidence)
4. Failure feeds back through the same gap-detection loop that DD uses

The convergence loop doesn't care that the "hypothesis" is executable code instead of a factual claim — it's just another fact with confidence, provenance, and a verification gate. Axiom already has the full pipeline (Gherkin → predicates → Rust → Wasm), and Helm owns the application plugin runtime that can host verified modules above Converge. The missing bridge is the `WasmSuggestor` adapter that would let verified Wasm modules *run* inside subsequent convergence loops without putting Wasmtime back into Converge.
